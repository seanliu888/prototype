# 客户及供应商结算规则说明

## 适用范围

本文说明系统自动结算任务生成客户账单和供应商账单的业务规则，面向运营、财务和业务使用者。文中不展开代码实现，只解释结算对象、触发时机、金额计算、账单状态和异常处理口径。

系统结算分为两条方向：

- 客户销售结算：面向客户，账单归属客户，结算方向为 `SALE`。
- 供应商成本结算：面向供应商，账单归属供应商，结算方向为 `COST`。

系统支持两类计费方式：

- 套餐计费：按订单单元和购买数量一次性结算。
- 流量计费：按实际用量结算，默认以 MB 记录用量，按 1024 MB = 1 GB 折算金额。

## 自动结算时间

系统每天凌晨自动执行三类结算任务：

| 任务 | 默认执行时间 | 处理内容 |
| --- | --- | --- |
| 套餐订单结算 | 凌晨 | 同时处理客户套餐销售结算和供应商套餐成本结算 |
| 客户流量销售结算 | 凌晨 | 处理客户侧按流量计费的销售账单 |
| 供应商流量成本结算 | 凌晨 | 处理供应商侧按流量计费的成本账单 |

如果业务需要，也可以通过管理接口手动触发结算，并可按客户或供应商过滤处理范围。

## 手动结算

除凌晨自动结算外，使用者也可以手动触发订单结算。手动结算会按相同规则依次处理：

- 套餐订单结算。
- 客户流量销售结算。
- 供应商流量成本结算。

手动结算不是重新计算所有历史账单，而是继续处理当前仍处于待结算状态的订单单元和用量记录。已经成功结算的记录不会重复生成账单。

手动结算适用于以下场景：

- 价格快照、用量或订单状态修复后，需要尽快重新处理。
- 运营或财务需要在自动任务前提前生成账单。
- 只想处理某个客户或某个供应商范围内的待结算数据。

手动结算完成后，系统会刷新受影响账期的汇总金额。返回的处理数量表示本次成功进入结算处理的记录数，不等同于账单总条数或总金额。

## 账期归属规则

结算账单按月归集。系统会按“账单归属对象 + 结算方向 + 币种 + 结算月份”生成结算周期。

| 结算对象 | 账单归属 | 结算方向 | 账期日期来源 |
| --- | --- | --- | --- |
| 客户套餐销售 | 客户 | `SALE` | 订单单元创建时间；如果为空，依次使用更新时间、接收时间、当前时间 |
| 供应商套餐成本 | 供应商 | `COST` | 订单单元创建时间；如果为空，依次使用更新时间、接收时间、当前时间 |
| 客户流量销售 | 客户 | `SALE` | 用量日期 |
| 供应商流量成本 | 供应商 | `COST` | 用量日期 |

示例：某客户在 2026-06-10 产生一笔客户套餐销售结算，币种为 USD，则该明细进入该客户 2026-06 的 USD 销售账期。

## 套餐结算规则

套餐结算只处理已经完成的订单单元。这里的“完成”指套餐已使用、已激活或超时未使用等情况，本质上是供应商侧订单已经进入不可退订状态。订单未完成、已结算或不是套餐计费的记录不会进入套餐结算。

### 客户套餐销售

客户套餐销售满足以下条件时会生成结算明细：

- 订单单元状态为已完成。
- 客户侧计费方式为套餐计费。
- 客户侧套餐结算状态为待结算。
- 存在该订单单元的客户销售价格快照。

金额计算：

```text
结算金额 = 客户销售单价 * 数量
```

数量为空时按 1 处理。价格使用下单时保存的价格快照，不读取当前价格配置，因此后续价格调整不会影响历史订单结算。

付款状态：

- 预付客户：`NOT_REQUIRED`。套餐销售通常已在订单流程中完成冻结或扣款，结算明细只作为账单记录。
- 后付客户：`UNPAID`。表示已入账但尚未收款，后续由收款或账期结算流程处理。

### 供应商套餐成本

供应商套餐成本满足以下条件时会生成结算明细：

- 订单单元状态为已完成。
- 供应商侧计费方式为套餐计费。
- 供应商侧套餐结算状态为待结算。
- 存在该订单单元的供应商成本价格快照。

金额计算：

```text
结算金额 = 供应商成本单价 * 数量
```

供应商侧账单付款状态统一为 `UNPAID`，表示待付款。系统当前不为供应商维护余额账户，实际付款完成后再由后续财务流程更新付款状态。

## 流量结算规则

流量结算基于每日用量记录。系统只处理结算状态为待结算的用量记录，并要求对应订单单元已经完成且计费方式为按流量计费。

流量金额统一按以下公式计算：

```text
金额 = 用量(MB) / 1024 * 每 GB 单价
```

客户销售价和供应商成本价都来自订单价格快照，并按供应商运营商维度匹配。也就是说，同一个订单单元在不同运营商下可以使用不同的流量单价。

### 客户流量销售

客户流量销售生成的是“增量明细”。系统会记录该订单用量之前已经结算到的水位，只对新增用量入账。

示例：

| 用量快照 | 已结算水位 | 本次结算用量 |
| --- | --- | --- |
| 500 MB | 0 MB | 500 MB |
| 800 MB | 500 MB | 300 MB |
| 800 MB | 800 MB | 不再生成新明细 |

金额口径：

- 系统先按当前累计用量计算应收总额。
- 再扣减历史已确认的客户流量销售金额。
- 差额大于 0 时，生成本次增量结算明细。

这样可以减少多次小额增量分别四舍五入带来的尾差问题。

付款状态：

- 预付客户：如果本次增量金额大于 0，系统会立即调用支付服务确认扣款；扣款成功后明细为 `PAID`。如果没有发生实际扣款，则为 `NOT_REQUIRED`。
- 后付客户：不立即扣款，明细为 `UNPAID`，后续由收款或账期结算流程处理。

### 供应商流量成本

供应商流量成本维护的是“累计明细”。同一供应商、同一订单单元、同一运营商、同一用量日期只保留一条成本明细；当用量增加时，系统更新这条明细为最新累计用量和最新累计成本。

示例：

| 用量快照 | 成本明细表现 |
| --- | --- |
| 500 MB | 创建一条 500 MB 的成本明细 |
| 800 MB | 更新同一条成本明细为 800 MB |
| 800 MB | 不产生变化 |

金额口径：

```text
供应商成本金额 = 当前累计用量(MB) / 1024 * 每 GB 成本价
```

付款状态统一为 `UNPAID`，表示待付款。

## 价格快照缺失时的处理

如果流量结算时找不到对应价格快照，系统不会直接把用量标记为已结算，而是生成或更新一条异常结算明细，方便运营或财务发现问题。

异常明细的特点：

- 账单状态为 `PENDING`。
- 付款状态为 `FAILED`。
- 单价和金额为 0。
- 备注会标明缺少客户销售价格快照或供应商成本价格快照，并记录运营商信息。
- 原用量仍保持待结算，价格修复后可在下一轮结算继续处理。

套餐结算如果缺少价格快照，则不会生成明细，也不会把订单单元标记为已结算，需要补齐价格快照后重新结算。

## 状态说明

### 订单或用量结算状态

| 状态 | 含义 |
| --- | --- |
| `PENDING` | 待结算，后续自动任务或手动触发会继续处理 |
| `SETTLED` | 当前订单单元或用量快照已完成结算处理 |
| `NOT_REQUIRED` | 该方向不需要进行流量结算，例如套餐计费方向不需要按流量入账 |

### 账单状态

| 状态 | 含义 |
| --- | --- |
| `PENDING` | 待确认或异常明细，常见于价格快照缺失 |
| `CONFIRMED` | 已确认入账，可参与账期汇总 |
| `REVERSED` | 已冲销 |

### 付款状态

| 状态 | 含义 |
| --- | --- |
| `NOT_REQUIRED` | 无需后续付款动作，常见于预付套餐或无实际扣款的预付流量记录 |
| `UNPAID` | 待收款、待扣款或待付款 |
| `PAID` | 已完成资金动作 |
| `FAILED` | 付款或账单异常，需要人工处理或补充数据 |

## 账期汇总规则

每轮结算完成后，系统会刷新受影响账期的汇总金额。汇总字段包括：

- 总金额。
- 套餐金额。
- 流量金额。
- 已结算流量。

账期汇总以结算明细为准，只统计有效明细。使用者查看账期金额时，应以账期汇总和明细列表共同核对；如果出现 `PENDING` 或 `FAILED` 明细，应先处理异常再确认账期。

## 手动扣款说明

手动扣款用于处理客户侧已经入账但尚未收款的结算明细。它只支持客户账单，不处理供应商成本账单。

### 账期级扣款

账期级扣款会处理指定客户账期下所有付款状态为 `UNPAID` 的结算明细。系统会分批查找未支付明细，并逐条执行扣款或标记已支付。

适用场景：

- 后付客户账期结束后，需要对整个月账单统一扣款。
- 某客户账期内有多条未支付明细，需要批量处理。

处理规则：

- 只支持客户账期，即账单归属为客户的账期。
- 只处理付款状态为 `UNPAID` 的明细。
- 金额大于 0 的明细，会从客户余额账户扣款，并生成余额交易流水。
- 金额小于或等于 0 的明细，不生成扣款流水，但会直接标记为 `PAID`。
- 已经是 `PAID`、`NOT_REQUIRED` 或 `FAILED` 的明细不会被账期级扣款处理。
- 如果客户余额账户不可用，或扣款后余额会超过允许的信用额度，扣款会失败，明细不会被标记为已支付。

账期级扣款结果通常关注以下数量：

- 本次处理的未支付明细数。
- 实际生成扣款流水的明细数。
- 零金额或非正金额、直接标记已支付的明细数。
- 本次实际扣款总金额。

### 明细级扣款

明细级扣款用于对单条结算明细执行扣款。它适用于只想处理某一笔客户应收明细的场景。

处理规则与账期级扣款一致：

- 只支持客户结算明细。
- 明细必须归属于对应客户账期。
- 明细付款状态必须为 `UNPAID`。
- 金额大于 0 时生成余额交易流水。
- 金额小于或等于 0 时直接标记为 `PAID`，不生成扣款流水。

如果明细不存在、不是客户明细、已经支付、无需支付、支付失败或归属账期不匹配，系统会拒绝本次扣款。

### 扣款与付款状态

手动扣款成功后，结算明细付款状态会更新为 `PAID`。金额大于 0 的明细会记录余额交易流水号，方便后续与客户余额流水对账。

扣款按结算明细维度保持幂等。同一笔明细如果已经生成过扣款流水，再次触发时不会重复扣款。

供应商侧账单的 `UNPAID` 表示待付款，但不通过这里的扣款能力处理。供应商付款仍由线下或独立财务流程完成，完成后再更新付款状态。

## 对账使用建议

- 客户侧对账时，优先按客户、月份、币种、`SALE` 方向查看账期。
- 供应商侧对账时，优先按供应商、月份、币种、`COST` 方向查看账期。
- 套餐类账单按订单单元核对数量和单价。
- 客户流量类账单按增量明细核对，每条明细代表某次新增用量产生的应收金额。
- 供应商流量类账单按累计明细核对，同一日期同一运营商通常只看最新一条明细金额。
- 发现价格快照缺失时，先补齐价格快照，再重新触发结算。
- 后付客户和供应商的 `UNPAID` 不代表结算失败，只表示资金动作尚未完成。
- 客户手动扣款后，应同时核对结算明细付款状态和客户余额交易流水。
